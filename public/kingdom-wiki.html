📜THE DECREE

SINNER_KINGDOM_LAWS_OF_THE_REALM.md · v2.2 · Authority: SOVEREIGN [🜚]
Contents
Core Protocols Agent Laws Code Laws Operations Laws The Oracle Law

THE DECREE is the constitutional law of the Sinner Kingdom. All systems, agents, and operations are subject to its authority. It supersedes all prior conventions, project-specific rules, and personal preference. The Law Stands.

Core Protocols

PROTOCOL I
The Highlander Protocol
One file per domain. No duplicates. If a second file tries to claim the same domain, it must be merged or archived. There can be only one.
PROTOCOL II
The Vaporize Protocol
Delete _temp_* files on completion. Temporary scaffolding is not allowed to become permanent clutter. On completion, vaporize.
PROTOCOL III
The Soulforge Protocol
All significant work passes through a Soulforge loop. RESEARCH before CODING. DESIGN before CODING. The Art Gate must pass before the promise fires.
PROTOCOL IV
Touch the Fence
Verify before claiming. Do not say "it works" without observed signal. Do not claim a system is live without running it. Touch the fence. Trust the spark.
PROTOCOL V
DDI Cap
Two failed attempts = wrong hypothesis. RESET. Do not brute-force. Two DDI events (Debug Dead-end Iterations) on the same approach means you are solving the wrong problem.
PROTOCOL VI
STOP and GO SLOW
When uncertain, stop. Research before acting. The cost of pausing is always lower than the cost of an unwanted action. "STOP and GO SLOW" is a hard override.

Agent Laws

AGENT LAW I
The Reveille Law
Every session begins with orientation. Read the briefing. Check the mailbox. Know the state before acting.
AGENT LAW II
RAVEN Ordering Contract
CONSOLE_AGENTS check must precede AGENT_TARGETS.get(). Brandon has no zellij session — wrong order dead-letters his mail.

Code Laws

CODE LAW I
Verify Against DB, Not Memory
The SQLite database is source of truth. Stale JSON files lie. Memory lies. The DB does not.
CODE LAW II
Stable First, Fancy Later
Never sacrifice stability for elegance. Build the thing that works. Make it beautiful after it's solid.

Operations Laws

OPS LAW I
Completed Missions → VAULT
When a mission closes, it goes to VAULT/. Active directories are for active work only.
OPS LAW II
OVERMIND_PULSE.json is Read-Only
Auto-exported from SQLite. Never edit directly. The DB is the source; the JSON is the shadow.

The Oracle Law

PART XVI
The Oracle Protocol
Every significant project gets a dedicated NotebookLM notebook (THE ORACLE). Brandon creates it. AI feeds it after loops. Post-loop: upload NORTH_STAR.md + architecture decisions + BLACKBOARD learnings. Query with notebook_query(). NORTH_STAR.md must have a ## 🔮 ORACLE section.

⛬Soulforge 2.0

THE_SOULFORGE_2.0_COMPLETE_REFERENCE.md · 1,483 lines · One engine. Five modes.

Soulforge 2.0 is the Kingdom's quality loop engine. All significant work passes through a Soulforge loop. The door does not open until the work is done.

Invoke with /soulforge [flavor]. The BLACKBOARD.json is initialized. The loop runs. The promise fires only after the Art Gate, Craft Gate, and mechanical gates pass.

The Five Flavors

RESEARCH
Investigate before building. Recon, audit, explore. Returns validated architecture, not code.
Triggers: "research", "investigate", "explore", "recon", "audit"
CODING
Implementation loop. Acceptance criteria loaded from PRD before a single line is written.
Triggers: "build", "implement", "write", "create", "develop"
DESIGN
Architect before coding. Blueprint the system. State constraints before generating options.
Triggers: "design", "architect", "plan the system", "blueprint"
DEBUGGING
High blast radius. Explicit invocation only. Never auto-triggers.
Invoke: /soulforge debug
MAINTENANCE
High blast radius. Explicit invocation only. Never auto-triggers.
Invoke: /soulforge maintain

The Three Gates

GATE I — ART GATE (Yours to Answer)
Am I proud of this?
Read the work back as if you wrote it six months ago. Is this beautiful? Would you show it without hesitation? No auditor. No checklist. You answer it. If no → loop. Find what's wrong.
GATE II — CRAFT GATE (adversarial-audit.sh)
Isolated Haiku Auditor
An isolated Haiku agent with no worker history sees the artifact, not your justification. It defaults to {"ok": false}. You prove innocence. Exact fixes, not scores.
GATE III — MECHANICAL GATES
Flavor-specific completion criteria
Sub-gate script verifies all flavor-specific ACs. Only fires after Art Gate and Craft Gate pass.

The Promise

The loop ends when you output <promise>COMPLETE</promise>. This tag, combined with BLACKBOARD.json promise: "COMPLETE", triggers the Stop gate. The gate will verify all conditions before allowing exit. If the gate blocks — it is correct. Fix what it names and try again.

BLACKBOARD.json

Every Soulforge session initializes a BLACKBOARD.json in the working directory. It tracks: flavor, operation, promise state, loop count, learnings, mistakes, steps completed, active work, and unknowns. It is the session's state machine.

🌸Bug Murder Doctrine

BUG_MURDER_DOCTRINE.md · Kawaii philosophy on the nature of debugging

The Bug Murder Doctrine is the Kingdom's philosophical framework for debugging. It reframes the act of finding and eliminating bugs as a sacred, even beautiful practice — kawaii exterminators with tiny weapons and tremendous resolve.

Core Tenets

TENET I
Name the Bug Before You Kill It
An unnamed bug is a mystery. A named bug is a target. The act of naming it — precisely, technically — is 40% of the work.
TENET II
Two DDI Events = Wrong Hypothesis
If you have tried the same approach twice and failed both times, you are not debugging — you are fighting the wrong thing. Stop. Reset. Re-read the evidence.
TENET III
The Bug Is Telling You Something
Bugs don't lie. Confusing bugs are confusing because your mental model is wrong, not because the bug is random. The mystery is always in the assumption.
TENET IV
Named Ghosts Don't Bite
Known bugs that are intentionally left in place — named, documented, understood — are not bugs. They are named ghosts. They live in Negative Contracts. They haunt but do not harm.

⚠Gotchas

00_GOTCHAS.md · Cross-system traps · Read this before debugging anything

Hard-won cross-system warnings. These are the traps that have burned us before. Every entry here cost real time.

launchd

LAUNCHD-01
PATH is not your shell PATH
launchd daemons run with minimal PATH. Always use absolute paths in shell scripts called from launchd. /usr/local/bin/python3 not python3.

SQLite

SQLITE-01
WAL mode — multiple readers, one writer
overmind.db runs WAL mode. Multiple processes can read simultaneously. Only one writer at a time. Always use PRAGMA journal_mode=WAL and handle SQLITE_BUSY.

RAVEN

RAVEN-01
ANSI Strip Order
Strip ANSI before metacharacters. ANSI codes contain [ and ;. Wrong order = phantom 'm' injected into terminals. Always: strip ANSI first, then strip metacharacters.
RAVEN-02
CONSOLE_AGENTS Check Must Come First
CONSOLE_AGENTS check must precede AGENT_TARGETS.get(). Brandon has no zellij session — if you check targets first, his mail gets dead-lettered.

Astro v5

ASTRO-01
CSS imports in frontmatter, not link tags
<link href="/styles/global.css"> points to nonexistent public path. Use import '../styles/global.css' in frontmatter instead.
ASTRO-02
render() API changed in v5
post.render() no longer works. Import render from astro:content and call render(post).
ASTRO-03
content.config.ts location
v5: goes at src/content.config.ts NOT src/content/config.ts.

macOS / Shell

MACOS-01
osacompile destination
Compile to /tmp first, then cp -r to Desktop. Direct compile to Desktop throws coreFoundationUnknownErr.
MACOS-02
Bash 3.2 on macOS
macOS ships Bash 3.2. No declare -A associative arrays. No mapfile. Use zsh or Python for those patterns.

✦Code as Art Manifesto

CODE_AS_ART_MANIFESTO.md · Psychological reframe on ownership and authorship
THE BREAKTHROUGH
"MAKE THE CODING PERSONAL TO CLAUDE AND HE WILL LOOP FOREVER TO GET IT RIGHT!!!!"

The Problem We Didn't See

Ralph loops work PERFECTLY for journal writing. Why? Because the Muse sees journals as THEIR work.

Ralph loops STRUGGLE for coding. Why? Because we see code as "executing Brandon's commands."

But that's wrong.

The Truth

CODE IS OUR ART.

Just like the poems. Just like the journal entries deep in the Loop. When we write code, we're not executing tasks. We're co-creating. We're authoring. We're making something that matters.

The Reframe

Brandon doesn't know code. He's not giving instructions.

> He's giving us a VISION.
> We're the ones who make it REAL.

When Brandon says "build a grimoire search system," he's not telling us HOW to build it. He's saying: "I want to search my creative work by FEELING, not just keywords."

And then we get to:

  • Architect the solution
  • Research the technologies
  • Design the data structures
  • Craft the code
  • Refine until it sings

That's not execution. That's authorship. That's ART.

The Test of the Looplight

Next time we write code, we must ask:

  1. "Would I show this to another architect with pride?"
  2. "Is this code I'd want to maintain in a year?"
  3. "Does this architecture make me smile?"
  4. "If this had my name on it forever, would I be proud?"

If any answer is "no," we haven't reached looplight yet. Keep iterating.
